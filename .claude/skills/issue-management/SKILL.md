# Issue Management Skill

## 概要

GitHub Issueを作成・整理・本文レビューするときに、タイトル、本文、記載者表記、レビュー運用を揃えるためのスキルです。

## 基本方針

- Issue本文は、現在有効な合意事項・目的・受け入れ条件を保つ場所として扱う。
- 古い前提、却下された案、不要になった説明、現在の受け入れ条件と矛盾する記述は、本文から削除するか現在の内容へ更新する。
- 判断の経緯や却下案は、本文ではなくコメント履歴に残す。
- Issue運用の詳細はこのSkillに置き、`CLAUDE.md` / `AGENTS.md` には共通原則とSkillへの導線だけを残す。
- Claude版とCodex版のIssue運用Skillで、タイトル接頭辞・本文テンプレート・記載者表記・本文レビュー運用・本文メンテナンス方針を揃える。

## 使用タイミング

- Issueを新規作成する時
- 既存Issueの本文を整理・統合・詳細更新する時
- Issue本文レビューを行う時
- Issueタイトル、本文、受け入れ条件、記載者表記、関連PRとの紐付けを確認する時

## Issueタイトル接頭辞

Issueタイトルには、関心事が分かる接頭辞を付ける。迷った場合は、Issueでユーザーに伝えたい主目的を優先して選ぶ。

- `[docs]:` ドキュメント、Skill、AGENTS/CLAUDE、運用ルール
- `[feat]:` 新機能、API追加、ユースケース追加
- `[bug]:` 不具合、期待挙動との差分、回帰
- `[refactor]:` 振る舞いを変えないコード整理、責務分離、構成変更
- `[test]:` テスト追加・改善、検証強化
- `[chore]:` 依存更新、設定調整、開発環境、CI以外の雑務
- `[ci]:` GitHub Actions、lint/typecheck/testなどのCI設定
- `[db]:` Prisma schema、migration、seed、DB制約
- `[security]:` 認証認可、秘密情報、脆弱性対応
- `[question]:` 方針相談、未確定事項の整理、調査

調査や技術検証は、独立した接頭辞を増やさず、まずは `[question]:` に寄せる。

## 本文テンプレート

Issue本文は固定テンプレートを重くしすぎず、以下の必須見出しを共通骨格とする。

```md
## 概要

<!-- Issueの背景・目的・やりたいことを書く -->

<!-- ここに関心事ごとの自由記載を追加する -->
<!-- 例: 背景 / 現状 / 課題 / 方針 / 非対象 / 再現手順 / API案 / 影響範囲 など -->

## 受け入れ条件

- [ ] 完了と判断できる条件を書く

## 記載者

Created by: <起票者>
Updated by: <本文を詳細更新したエージェント>
```

- `概要`、`受け入れ条件`、`記載者` は必須とする。
- 関心事ごとの自由記載は `概要` と `受け入れ条件` の間に置く。
- 自由記載の見出しはIssueの内容に応じて必要なものだけ追加する。
- `受け入れ条件` は、Issueを閉じる判断に使える粒度のチェックリストにする。
- `記載者` は本文末尾に置き、誰が本文を書いたか・誰が本文を更新したか分かるようにする。

## 記載者表記

記載者ブロックは、`Created by:` と `Updated by:` の積み上げ形式にする。

- `Created by:` は起票者を1行で記載し、以後変更しない。
- `Updated by:` は本文を詳細更新したエージェントを行で積み上げる。
- 同一エージェントの `Updated by:` は重複して追加しない。
- 起票者がユーザーの場合、ユーザー自身は記載者ブロックを書かなくてよい。
- Claudeが本文を詳細更新するときはIssueのauthorを確認し、起票者がリポジトリ所有者なら `Created by: me` を補完したうえで、`Updated by: Claude <claude@anthropic.com>` を追記する。
- 過去Issueへの一括適用は必須にせず、新しく作成・整理するIssueから適用する。

表記例:

```md
## 記載者

Created by: me
Updated by: Codex <codex@openai.com>
Updated by: Claude <claude@anthropic.com>
```

起票者別の `Created by:` は以下に揃える。

- ユーザー本人が起票したIssue: `Created by: me`
- Codexが起票したIssue: `Created by: Codex <codex@openai.com>`
- Claudeが起票したIssue: `Created by: Claude <claude@anthropic.com>`

## 作成・整理手順

1. 既存Issue、関連コメント、関連PRがある場合は先に読む。
2. タイトル接頭辞が関心事と合っているか確認する。
3. 本文に `概要`、必要な自由記載、`受け入れ条件`、`記載者` があるか確認する。
4. `受け入れ条件` が完了判断に使える粒度か確認する。
5. 本文を詳細更新する場合は、現在有効な内容へ整理し、古い前提や却下案は本文に残さない。
6. 記載者ブロックを確認し、必要なら `Created by:` を補完し、自分の `Updated by:` を重複なしで追記する。
7. 大きく整理した場合は、コメントで変更点と理由を残す。

## コメント記法

Issueへコメントを投稿するときは、コメントの関心事と、コメントしたエージェントを一目で判別できるようにする。

- コメントの先頭に、関心事を示す大見出し（h2相当）を置く。
- 大見出しの内容は、そのコメントの関心事にする（例: レビュー結果 / 対応内容 / 調査結果 など）。
- 大見出しの横に、コメントしたエージェント名を併記する。
- 形式: `## <コメントの関心事> (<エージェント名>)`
  - 例: `## レビュー結果 (Claude)` / `## 調査結果 (Codex)`

## 本文レビュー運用

- レビュワーはIssue本文を直接変更・更新しない。
- レビュワーはコメントで意見や本文への変更依頼箇所を明示する。
- 変更依頼コメントでは、対象見出し、変更理由、必要なら修正文案を記載する。
- 本文の更新は、作成者または明示的に引き継いだ担当者が行う。
- 本文を更新した担当者は、対応したレビューコメントへ返信する。
- 問題がなければ、レビュワーは `:+1:` を含むコメントで問題ない旨を記載する。

## ラベル・担当者・milestone・project

- ユーザーから指定がある場合は、その指定を優先する。
- 明確な既存運用がある場合だけ、ラベル、assignee、milestone、projectを設定・更新する。
- 判断に迷う場合は、Issue本文やコメントで意図を確認してから変更する。
- Claudeが自動判断で不要なメタデータ変更を広げない。

## IssueとPRの紐付け

- 対象IssueがあるPRでは、PR本文の `関連Issue` セクションに `Closes #<issue番号>` / `Fixes #<issue番号>` / `Resolves #<issue番号>` のいずれかを記載する。
- 対象Issueがない場合は `なし` と記載し、空の `Closes #` を残さない。
- PR本文の `概要` / `変更内容` / `確認内容` は、対象Issueの受け入れ条件と矛盾しないように更新する。
- Issue側の受け入れ条件チェックを更新する場合は、PR差分・テスト・動作確認により満たしたと判断できる項目だけに限定する。

## Claudeでの実行メモ

- Issueの取得・作成・更新・コメント投稿は、`gh issue view`、`gh issue create`、`gh issue edit`、`gh issue comment` などを使う。
- Issue本文レビューでは、まず本文とコメントを取得し、既に合意済みの変更依頼が反映されているか確認する。
- ClaudeがIssueを新規作成・詳細整理した場合は、本文末尾の記載者表記とコメントで作業主体が分かるようにする。
