# Review Response Skill

## 概要

GitHub Pull Requestに入ったレビュー指摘へ対応し、修正・確認・PR本文更新まで行うスキルです。

レビューを行う側の手順は `.claude/skills/pr-review/SKILL.md` を使用し、このSkillではレビューに対応する側の作業に集中します。

## 使用タイミング

- ユーザーが「レビュー対応して」と依頼した時
- ユーザーがPR番号やURLを指定して、レビュー指摘への対応を求めた時
- レビューコメント対応後に、PR本文・差分・コミット履歴の整合性確認を求められた時

## 正本と責務分担

- レビュー対応コミット形式と、レビュー対応時のPR本文更新手順の詳細は、このSkillを正本とする。
- [CLAUDE.md](../../../CLAUDE.md) の `#### レビュー指摘への対応コミット` / `#### レビュー対応時のPR本文更新` には、Skillを起動しなくても常に守る共通原則（レビュー対応はPR上の追加コミットで行う、PR本文更新はレビュー対応フローの一部、など）だけが置かれている。共通原則とこのSkillの内容がずれないようにすること。
- Skill化していない運用ルール（コミット粒度、PRタイトル、ブランチ命名、push方法）は、引き続き [CLAUDE.md](../../../CLAUDE.md) を正本とし、必要な箇所で参照する。

## 実行手順

### 1. PRとレビューコメントの確認

```bash
gh pr view <PR番号またはURL>
gh pr checks <PR番号>
gh pr diff <PR番号>
```

- 対象PRの番号、base/head branch、最新HEADを確認する
- CI状態を確認する
- 未解決のレビューコメント、レビュー本文、通常コメントを確認する
- 指摘がインラインコメントの場合は、対象ファイルと行を確認する

### 2. 対応方針の整理

- 指摘ごと、または関心事ごとに対応単位を分ける
- 修正が必要な指摘と、説明・コメント返信で足りる指摘を分ける
- 指摘同士が衝突する場合は、作業前にユーザーへ確認する

### 3. 修正の実装

- 既存のアーキテクチャ、features設計、命名規則に合わせて修正する
- 不要なリファクタリングを混ぜない
- レビュー指摘に対応する変更だけを行う

### 4. 確認コマンドの実行

変更内容に応じて必要な確認を実行する。

```bash
bun run spellcheck
bun run lint
bun run typecheck
bun test
bun run build
```

ドキュメントのみの変更でも、少なくとも `bun run spellcheck` を確認する。

### 5. レビュー対応コミットの作成

レビュー指摘へ対応する追加コミットは、通常のコミットと区別できるように以下の形式で記述する。

```txt
review: 作業内容

comment: レビューコメントのURL

Co-authored-by: 作業したAIエージェントの名前 <メールアドレス>
```

- `review:` の後には、対応した作業内容を日本語で簡潔に記述する
- `comment:` には、対応元のGitHubレビューコメントまたはレビューのURLを記述する
- AIエージェントが対応した場合は、通常のコミットと同様に `Co-authored-by` トレーラーを付与する（Claudeなら `Co-authored-by: Claude <claude@anthropic.com>`）
- 指摘が複数ある場合は、指摘ごと、または関心事ごとにコミットを分ける
- 各コミット直前に `.claude/skills/commit-granularity/SKILL.md` をSkillとしてinvokeし、「コミット直前チェック」でステージ済み差分の粒度を確認する

### 6. PR本文の更新

push前またはpush直後に、PR本文を読み直し、最新状態へ更新する。PR本文の更新は、コード修正とは別の補助作業ではなく、レビュー対応フローの一部として扱う。

少なくとも以下を確認する。

- `概要` が現在のPR内容と一致していること
- `変更内容` がレビュー対応分を含めて最新であること
- `コミット構成` が現在のPRの状態を正しく説明していること
- `確認内容` のチェック状態が最新であること
- レビュー対応コミットを1つ以上追加した場合は、PR本文に `レビュー対応` セクションを必ず追加・更新すること（任意ではなく必須）。このセクションには、レビュー指摘を受けて具体的に行ったこと、または対応しない判断をしたことを、レビュー担当者が追える粒度でリスト形式で記載すること。このPR本文の `レビュー対応` セクションは、ステップ8で投稿するPR Conversationの `## レビュー対応 (Claude)` コメントとは別物であり、どちらか一方で他方を代替しない
- PR本文と実際の差分・コミット履歴にズレがないこと

`コミット構成` は、マージ方式によって最終的なコミット数が変わる可能性があるため、厳密な最終履歴との一致よりも、レビュー時点のPRの状態を正しく説明していることを優先する。

確認には必要に応じて以下を使う。

```bash
gh pr diff <PR番号>
gh pr view <PR番号> --json commits
```

### 7. pushと最終確認

```bash
git push <push先> <branch>
gh pr checks <PR番号>
```

- push後にCI状態を確認する
- PR本文、差分、コミット履歴が現在のPR状態を正しく説明しているか確認する
- レビュー対応コミットを1つ以上追加した場合は、PR Conversationへ対応内容コメントを必ず投稿する（手順はステップ8を参照）

### 8. レビュー対応コメントの投稿

レビュー対応コミットを1つ以上追加した場合は、PR Conversationへ対応内容コメントを必ず投稿する。これはレビュー対応フローの必須手順であり、省略しない。

このPR Conversationの対応内容コメントと、ステップ6で更新するPR本文の `レビュー対応` セクションは別物として扱う。

- PR本文の `レビュー対応` セクションを追加・更新しても、PR Conversationへの対応内容コメント投稿の代替にはしない。
- PR Conversationへ対応内容コメントを投稿しても、PR本文の `レビュー対応` セクション追加・更新の代替にはしない。
- レビュー対応コミットを追加した場合は、両方を必ず行う。

対応内容コメントは以下に従って投稿する。

- 見出しは半角括弧の `## レビュー対応 (Claude)` に固定する。
- 対応元がインラインレビューコメントの場合、該当スレッドへの返信は必要に応じて行ってよいが、PR Conversationへの対応内容コメントは省略しない。
- 複数のレビュー対応コミットを追加した場合、コメントは1件にまとめてもよいが、各コミットの内容が追える粒度で記載する。
- 対応内容コメントには、少なくとも以下を含める。
  - 対応コミットの短縮SHAとコミット件名
  - 対応元レビューコメントまたはレビューのURL
  - 実装した対応内容
  - 実行した確認コマンドと結果
  - PR本文を更新したこと

コメント例:

```md
## レビュー対応 (Claude)

- `1a2b3c4` review: 〇〇への対応
  - comment: https://github.com/owner/repo/pull/NN#discussion_rNNNNN
  - 対応内容: 〇〇を修正した
  - 確認: `bun run spellcheck` / `bun run lint` がgreen
- PR本文の `レビュー対応` セクションを追加・更新済み
```

投稿には必要に応じて以下を使う。

```bash
gh pr comment <PR番号> --body-file <ファイル>
```

## 注意事項

- PR本文更新は、レビュー対応フローの一部として必ず扱う
- レビュー対応コミットを追加した場合は、PR Conversationへの対応内容コメント投稿とPR本文 `レビュー対応` セクション更新の両方を必須とし、どちらか一方で他方を代替しない
- Codex版（`.codex/skills/review-response/SKILL.md`）と方針・確認項目がずれないようにする
- レビュー対応コミット形式・PR本文更新手順の詳細はこのSkillを正本とし、CLAUDE.md側の共通原則とずれないようにする
- コミット粒度・PRタイトル・ブランチ命名・push方法など、Skill化していないルールはCLAUDE.mdを参照する
- `pr-review` Skillと混同せず、レビューする側ではなく対応する側の作業に使う
- コメント、ドキュメント、コミットメッセージは日本語ルールに従う
