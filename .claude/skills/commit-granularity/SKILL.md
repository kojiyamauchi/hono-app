# Commit Granularity Skill

## 概要

コミットの分解・作成・レビュー時に、リポジトリのコミット粒度ルールへ沿っているか確認するスキルです。

## 正本

- コミット粒度の正本は [CLAUDE.md](../../../CLAUDE.md) / [AGENTS.md](../../../AGENTS.md) の「コミットの粒度」「feature実装時の標準コミット粒度」です。
- このSkillは正本を再掲せず、実作業で確認するチェックリストに徹します。

## 使用タイミング

- 作業開始時に、今回の変更をコミット単位へ分解する時
- 各コミット直前に、ステージ済み差分の粒度を点検する時
- PRレビュー時に、コミット履歴がレビュー可能な粒度か確認する時
- レビュー指摘対応時に、指摘ごと・関心事ごとの追加コミットになっているか確認する時

## 作業開始時チェック

- 今回の作業は、PR本文の作業項目に対応する単位へ分解できているか
- APIやユースケース追加では、DTO / mapper / repository / service / schema / controller / routes / top-level route登録を混ぜず、レイヤー単位に分けているか
- 実装コミットと対応テストコミットを隣接させる計画になっているか
- 薄いcontrollerや単純なrepositoryなど、単体テストを省略する場合でも、serviceテストまたはroute統合テストで振る舞いを担保する計画になっているか
- migrationや非自明なデータ層を含む場合、実DB検証のタイミングと証跡を残す場所が決まっているか

## コミット直前チェック

`git diff --cached --stat` と `git diff --cached` を確認してから判断します。

- このコミットは1つの関心事に閉じているか
- feature標準レイヤーが不自然に混ざっていないか
- 特に `controllers/` と `routes/`、または `DTO + mapper + repository`、`service + schema + controller + routes` をまとめていないか
- 実装コミットの場合、対応するテストを同一コミットまたは直後のコミットで扱う計画が明確か
- レビュー対応コミットの場合、対応元コメントURLをコミット本文へ入れる必要があるか
- AIエージェントによるコミットの場合、適切な `Co-authored-by` トレーラーを付ける準備ができているか

## PRレビュー時チェック

- `git log --oneline <base>..HEAD` でコミット一覧を確認したか
- `git show --stat <commit>` で各コミットの変更範囲を確認したか
- 複数の関心事が混ざり、レビューや将来の追跡が難しいコミットがないか
- 実装とテストの隣接が崩れていないか
- review対応コミットが、指摘ごと、または関心事ごとに分かれているか

## 判定

- OK: 1コミットが1つの関心事に閉じ、実装とテストの関係が追える
- 要分割: 複数レイヤーや無関係な文書・設定変更が混ざり、レビュー単位として大きすぎる
- 要確認: 標準ディレクトリでは表現しづらい責務や、テスト省略の妥当性に設計判断が必要
